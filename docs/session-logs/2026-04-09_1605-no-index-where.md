# 요청 요약

- `main -> dev -> feat/no-index-where` 순서로 브랜치를 정리한 뒤
  full scan 기반 `WHERE`와 SQL 총 실행 시간 출력 기능을 추가

# 결정 사항

- 작업 전 로컬 변경은 `git stash push -u`로 분리
- 최종 작업 브랜치는 `feat/no-index-where`
- `WHERE`는 `SELECT` 전용으로 지원
- 비교 연산자는 `=`, `<`, `<=`, `>`, `>=`를 지원
- 조건은 최대 2개까지 `AND`로 연결
- `OR`는 명시적으로 미지원 처리
- 총 실행 시간은 `stderr`에 `총 실행 시간: <ms> ms` 형식으로 출력

# 현재 브랜치 상태

- 시작 브랜치: `main`
- `dev`에 `main`을 `--no-ff`로 병합
- 현재 브랜치: `feat/no-index-where`

# 완료한 작업

- `include/sqlproc.h`에 WHERE 관련 토큰/구조체/비교 연산자 추가
- `src/tokenizer.c`에 WHERE/AND/OR 및 비교 연산자 토큰화 추가
- `src/parser.c`에 `SELECT ... FROM ... [WHERE ...]` 파싱 추가
- `src/executor.c`에 WHERE 유효성 검증 추가
- `src/storage.c`에 full scan WHERE 필터링 추가
- `src/app.c`에 wall-clock 기반 총 실행 시간 출력 추가
- `tests/test_runner.c`에 WHERE/timing 회귀 테스트 추가
- `examples/where_demo.sql`, `README.md` 업데이트

# 리뷰 결과

- 정확성/버그
  - `WHERE`는 `SELECT` 전용으로 제한했고, 비교 연산자 5종과 `AND` 2개까지를 테스트로 검증
  - `WHERE` 타입 불일치, `OR`, 3개 이상 조건, 손상된 정수 CSV 값을 실패 케이스로 확인
- 자료구조 무결성
  - `SelectStatement`에 `WhereClause`를 추가하고, executor가 스키마/리터럴 검증을 먼저 수행
  - storage는 검증된 조건만 받아 full scan 비교를 수행하도록 역할을 분리
- 초심자 가독성/C99
  - 기존 tokenizer/parser/executor 흐름을 유지하면서 WHERE 관련 함수만 단계별로 추가
  - 실행 시간 출력은 `app.c`에 한곳으로 모아 SELECT 출력 형식을 건드리지 않음
